# PluPro v1.1 — 项目分析触发链路设计

> v1.1 在 v1 控制塔基础上增加"PM 一键标记 → Claude Code 一键拆解"两段链路。
> v1.1 不实施 PluPro 直接 spawn Claude CLI(留 P2),仅做"标记 + Notice + slash command"
> 的协作模型。
>
> 协议层(项目模板/协议地图/prompt 规范)已经在 P0 阶段落盘,本 spec 仅约束插件代码侧
> 与 vault `.claude/commands/` 侧的实施细节。

---

## 1. 范围

### 1.1 v1.1 in

- PluPro 插件:命令 `用 Claude 分析此项目`(命令面板可调,项目详情区也加按钮)
- PluPro 写回 frontmatter:`pending-analysis: true` + `last-analyzed: <ISO>`
- PluPro 控制塔项目卡片显示"待分析"/"已拆解"徽章
- HIC vault 内 slash command:`<vault>/.claude/commands/analyze-project.md`
- slash command 引用 P0 已落盘的 prompt 规范

### 1.2 v1.1 out(明确推迟到 P2+)

- PluPro 进程直接 spawn Claude CLI(P2)
- 偏差反馈循环(P3 — git log 扫描 + 偏差摘要面板)
- 拆解进度可视化(P2 — stdout 流式显示)
- 取消标记 / 重新分析等 frontmatter 反向操作(若有需要,在 v1.2 加)

## 2. 数据模型

### 2.1 复用 P0 模板已定义的字段

`_projects/<slug>.md` frontmatter:

```yaml
# v1.1 消费/写入的字段(P0 模板已含)
pending-analysis: false              # PluPro 写,初始 false
last-analyzed: null                  # Claude 拆解后写 ISO timestamp
generated-changes: []                # Claude 拆解后写 change slug 列表
```

v1.1 **不引入新 frontmatter 字段**。

### 2.2 状态机

```
[初始]                pending-analysis=false, generated-changes=[]
   │
   │ PM 调命令"用 Claude 分析此项目"
   ▼
[待分析]              pending-analysis=true,  last-analyzed=<标记时刻>
   │                  generated-changes=[](不变)
   │ Claude 跑 /analyze-project,拆解中
   │ Claude 创建 N 个 OpenSpec change(每个 proposal.md frontmatter 加 project: <slug>)
   │ Claude 回填 manifest
   ▼
[已拆解]              pending-analysis=false, last-analyzed=<完成时刻>
                      generated-changes=[change-slug-1, change-slug-2, ...]
```

**徽章 UI 映射**:
- `pending-analysis === true` → `🟡 待分析` 黄色 chip
- `pending-analysis === false && generated-changes.length > 0` → `🟢 已拆解(N)` 绿色 chip
- 其余 → 无徽章

## 3. 架构

### 3.1 PluPro 插件侧改动

#### 3.1.1 新增命令(`src/main.ts`)

```typescript
this.addCommand({
  id: 'mark-project-for-analysis',
  name: '用 Claude 分析此项目',
  checkCallback: (checking: boolean) => {
    const file = this.app.workspace.getActiveFile();
    if (!file || !this.isProjectManifest(file)) {
      return false;
    }
    if (!checking) {
      void this.markProjectForAnalysis(file);
    }
    return true;
  },
});
```

`checkCallback` 让命令仅在 `_projects/` 下文件激活时出现在命令面板。

#### 3.1.2 新增方法 `markProjectForAnalysis(file: TFile)`

```typescript
async markProjectForAnalysis(file: TFile): Promise<void> {
  const slug = await this.frontmatterIO.readField<string>(file, 'slug');
  const type = await this.frontmatterIO.readField<string>(file, 'type');

  if (type !== 'project' || !slug) {
    new Notice('当前文件不是有效项目 manifest(缺 type=project 或 slug)');
    return;
  }

  await this.app.fileManager.processFrontMatter(file, (fm) => {
    fm['pending-analysis'] = true;
    fm['last-analyzed'] = new Date().toISOString();
  });

  new Notice(
    `已标记 ${slug} 为待分析。\n在 Claude Code 内运行:/analyze-project ${slug}`,
    8000,
  );

  await this.refreshIndex();
}
```

Notice 显示 8 秒(默认 4 秒不够看 slash command 提示)。

#### 3.1.3 `isProjectManifest(file)` helper

```typescript
private isProjectManifest(file: TFile): boolean {
  return (
    file.path.startsWith(this.settings.projectsPath + '/') &&
    file.extension === 'md' &&
    !file.basename.startsWith('_')  // 排除 _template.md 等
  );
}
```

#### 3.1.4 ControlTowerView 项目详情区按钮

`renderDetail` 内,在 project title 后加按钮:

```typescript
const actions = this.detailPaneEl.createDiv({ cls: 'plupro-detail-actions' });
const analyzeBtn = actions.createEl('button', {

analyzeBtn.addEventListener('click', () => {

```

const fm = entry.manifest;  // 注意:需要 ProjectManifest 暴露 pending-analysis / generated-changes

### 3.2 HIC vault 侧:slash command

#### 3.2.1 文件位置

`<vault>/.claude/commands/analyze-project.md`

注:这是 vault-scoped slash command(仅在 HIC vault 内激活,不影响其他 vault)。

#### 3.2.2 文件内容

```markdown
---
description: 按规范分析项目并自动拆解为 OpenSpec changes
argument-hint: <project-slug>
---

读取项目 `_projects/<slug>.md`,按
`_meta/openspec-control-tower/project-to-changes-prompt.md`
中定义的标准 prompt 执行拆解。

参数 `$ARGUMENTS` 是项目 slug。

要点:
1. **必须读取完整 prompt 规范文件作为指导**,不要凭记忆执行
2. 项目文件可能是中文文件名(如 `瓶颈区域布线.md`),通过 frontmatter slug 字段匹配
3. 严格遵守 prompt 规范的命名约束 + 输出格式
4. 完成后必须回填 manifest:generated-changes / last-analyzed / pending-analysis: false
5. 每个新建的 OpenSpec change,proposal.md frontmatter 必须含 `project: <slug>`

如果 slug 不存在或项目文件 vision/goals 太模糊,**不要捏造 change**,
在 manifest body 加一段 `## ❓ Claude 提问` 列出需要 PM 澄清的点。
```

## 4. 核心流程

### 4.1 PM 触发分析(PluPro 侧)

```
1. PM 在 Obsidian 打开 _projects/<slug>.md
2. Cmd-P → "PluPro: 用 Claude 分析此项目"(或点击项目详情区的按钮)
3. PluPro 验证 type=project + slug 存在 → 不通过则 Notice 拒绝
4. PluPro 用 processFrontMatter 写入:
   - pending-analysis: true
   - last-analyzed: <当前 ISO>
5. PluPro 弹 Notice 8 秒:"已标记 X 为待分析。在 Claude Code 内运行 /analyze-project X"
6. PluPro refreshIndex,控制塔项目卡片立即显示 🟡 待分析 徽章
```

### 4.2 Claude 拆解(Claude Code 侧)

```
1. PM 在 Claude Code 对话里输入 /analyze-project bottleneck-routing
2. Claude 加载 vault 的 .claude/commands/analyze-project.md slash command
3. Claude 读取 _projects/<slug>.md(可能是中文文件名,按 slug 匹配)
4. Claude 读取 _meta/openspec-control-tower/project-to-changes-prompt.md 标准 prompt
5. Claude 读取 openspec/changes/ 已有 changes 清单
6. Claude 按 prompt 内部规划:每个 goal → 0/1/N 个 change
7. Claude 用 /opsx:propose 链路批量创建 change(每个 proposal.md frontmatter 含 project:)
8. Claude 用 processFrontMatter API(或文件写入)回填 _projects/<slug>.md:
   - pending-analysis: false
   - last-analyzed: <完成时 ISO>
   - generated-changes: [change-slug-1, change-slug-2, ...]
9. Claude 输出报告:"共生成 N 个 change,建议下一步..."
```

### 4.3 PluPro 自动反映(metadataCache 订阅)

```
1. Claude 写完 _projects/<slug>.md,触发 vault 'modify' / metadataCache 'changed' 事件
2. PluPro 已订阅这两个事件(v1 已实现 250ms debounce)
3. PluPro refreshIndex → 控制塔徽章从 🟡 切换为 🟢 已拆解(N)
4. 同时 PluPro 也 refresh changes(因为 Claude 在 openspec/changes/ 创建了新目录,触发 'create' 事件)
5. 控制塔项目下挂的 changes 数量从 0 变为 N
```

## 5. 错误处理

| 场景 | PluPro 行为 | Claude 行为 |
|------|------------|-------------|
| 命令在非 _projects/ 下文件调用 | checkCallback return false,命令不出现 | n/a |
| 文件 frontmatter 缺 type 或 slug | Notice "不是有效项目 manifest" | n/a |
| FrontmatterIO 写入失败 | Notice 显示错误,索引不变 | n/a |
| slash command 在非 HIC vault 内激活 | n/a | slash command 文件不存在,命令找不到 |
| slash command slug 参数无对应文件 | n/a | 报告"找不到 slug=X 的项目" |
| 项目 vision/goals 太模糊 | n/a | 不创建 change,在 manifest body 加"❓ Claude 提问"段落 |
| Claude 拆解失败(网络/API) | n/a | session 报错,manifest 状态保持"待分析"(pending-analysis 仍为 true),用户可重试 |

## 8. 已知 v1.1 范围限制

- **slash command 只在 HIC vault 内可用**(因为依赖 vault 内 `_meta/openspec-control-tower/` 的 prompt 规范)。其他 vault 用户需要自行复制 `_meta/openspec-control-tower/` 目录或重写 prompt。
- **不做拆解过程可视化**:Claude session 在 Claude Code 终端跑,PluPro 不流式显示进度,PM 需要切到 Claude Code 看。
- **不做拆解回滚**:如果 Claude 拆解错了,PM 需要手动删 OpenSpec change 目录 + 清 manifest 的 generated-changes 字段。
- **slug 匹配方式**:slash command 用 `$ARGUMENTS` 作为 slug,与 manifest 的 `slug` 字段匹配,**不**与文件名匹配。中文文件名 OK,只要 frontmatter slug 正确。

## 9. 已确认的关键决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| frontmatter vs body | frontmatter 主导 + body 可选 | P0 brainstorming 阶段选定,机器读 + Bases 友好 |
| 触发方式 | 命令面板 + 详情区按钮(标记)+ Claude slash command(拆解) | v1.1 不做 spawn,降低复杂度,先验证 prompt 准确性 |
| 状态机 | 三态:初始 / 待分析 / 已拆解 | 简单,徽章 UI 易实现 |
| slash command 位置 | vault scoped(`.claude/commands/`) | 仅在 HIC vault 内可用,符合"协议放 vault"原则 |
| 命令仅 _projects/ 激活 | checkCallback 路径前缀检查 | UX 防误调 |
| 排除 `_template.md` | file.basename 不能以 `_` 开头 | _template 不是真项目 |

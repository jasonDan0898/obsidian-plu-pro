## Task 2: main.ts 加 isProjectManifest helper + markProjectForAnalysis 方法

**Files:**
- Modify: `src/main.ts`

### Step 1: 加入 helper 和 method

在 `src/main.ts` 的 `openAssignmentModal(target: ChangeEntry): void {` 方法**之后**(class 内部),加入两个新方法:

```typescript
  /**
   * 检查文件是否为 _projects/ 下的有效项目 manifest 文件。
   * - 路径必须在 settings.projectsPath 下
   * - 必须是 .md
   * - 排除 _ 前缀的模板/工具文件(如 _template.md)
   */
  private isProjectManifest(file: TFile): boolean {
    return (
      file.path.startsWith(this.settings.projectsPath + '/') &&
      file.extension === 'md' &&
      !file.basename.startsWith('_')
    );
  }

  /**
   * 标记项目为"待分析",写回 frontmatter pending-analysis: true + last-analyzed,
   * 弹 Notice 提示用户在 Claude Code 跑 /analyze-project <slug>。
   */
  async markProjectForAnalysis(file: TFile): Promise<void> {
    const slug = await this.frontmatterIO.readField<string>(file, 'slug');
    const type = await this.frontmatterIO.readField<string>(file, 'type');

    if (type !== 'project' || !slug) {
      new Notice('当前文件不是有效项目 manifest(缺 type=project 或 slug)');
      return;
    }

    await this.app.fileManager.processFrontMatter(file, (fm) => {
      fm['pending-analysis'] = true;
      fm['last-analyzed'] = new Date().toISOString();
    });

    new Notice(
      `已标记 ${slug} 为待分析。\n在 Claude Code 内运行:/analyze-project ${slug}`,
      8000,
    );

    await this.refreshIndex();
  }
```

### Step 2: 创建 slash command 文件

File: `/Users/danwei/Documents/HIC/.claude/commands/analyze-project.md`

```markdown
---
description: 按规范分析项目并自动拆解为 OpenSpec changes
argument-hint: <project-slug>
---

你被调用了 `/analyze-project $ARGUMENTS`。参数 `$ARGUMENTS` 是项目 slug。

## 执行步骤

1. **读取标准 prompt 规范**(必须):
   读 `_meta/openspec-control-tower/project-to-changes-prompt.md`

2. **定位项目 manifest 文件**:
   - 在 `_projects/` 目录下查找 frontmatter `slug === $ARGUMENTS` 的 .md 文件
   - 文件名可能是中文(如 `瓶颈区域布线.md`),按 slug 字段匹配,不要按文件名

3. **读取项目内容**:
   读取该 manifest 完整内容(frontmatter + body),特别关注:
   - vision、goals、success-criteria、non-goals、scope、risks

4. **读取 OpenSpec 现状**:
   列出 `openspec/changes/` 下已有的 change(不含 archive/),避免重复创建

5. **拆解执行**(按 prompt 规范):
   - 严格遵守 prompt 规范的命名约束 + 输出格式
   - 对每个 goal 判断需要 0/1/N 个 change
   - 用 `/opsx:propose <kebab-slug>` 链路批量创建
   - 每个 change 的 proposal.md frontmatter **必须**含 `project: $ARGUMENTS`
   - 跑 `openspec validate <slug> --strict --no-interactive` 通过

6. **回填 manifest**(使用 Edit 工具):
   - `pending-analysis: false`
   - `last-analyzed: <当前 ISO timestamp>`
   - `generated-changes: [<新创建的 change slug 列表>]`

7. **输出报告**:
   - 共生成 N 个 change 的摘要(slug + title + 对应 goal)
   - 建议下一步:依赖关系/实施顺序

## 失败处理

- slug 不存在 → 报告"找不到 slug=$ARGUMENTS 的项目"
- vision/goals 太模糊 → **不要捏造**,在 manifest body 加 `## ❓ Claude 提问` 段落列出待澄清的点
- prompt 规范文件不存在 → 报告"本命令仅在 HIC vault 内可用,prompt 规范缺失"

## 禁止

- 不要凭记忆执行,**必须**先读 prompt 规范文件
- 不要捏造 capability 名,只用 openspec/specs/ 下真实存在的
- 不要把"测试" / "文档"单独立为 change
- 不要修改 manifest 的 vision/goals(那是 PM 的领地)
```
